# 基于工业级项目规范的AI辅助开发工作流重构计划

## 一、问题陈述

在使用Claude Code等AI工具辅助开发时，我发现一个现象：每次提出需求后，Agent能够直接完成一个功能级的代码。这对代码审计造成了极大挑战。即便Agent功能实现的不完全正确，其中的部分代码也并非完全没有意义。但是每次版本管理时，只能一次commit Agent一整轮的超大更新——这在以前会被拆为多个commit甚至一个完整分支。

一个具体的场景如下：

1. 我用Plan Mode规划了一个任务，给定输入和输出，让LLM去编码；
2. LLM返回了一个约500行的变更；
3. 但是我只有一组输入和输出，对于中间状态我根本都没有构想过它应该是什么样的；
4. 我拆不开那500行的变更，因为它们符合我的输入和输出，但是因为我把任务粒度规划得比较大，不知道拆解后如何独立验证。

此外，还存在两个深层次问题：

**上下文过载与过程不透明**：Agent在实现宏大需求时，可能在错误的路径上渐行渐远，而开发者无法在早期介入纠正。这源于三个根因：需求粒度过大、中间状态缺乏可验证性、开发者对Agent内部机制缺乏干预能力。

**测试信心的断崖**：在编译器开发过程中，我遇到了“抽象降级”的问题——我基于手写的、某种理想状态中的LLVM IR开发了功能并进行了测试，然而在端到端测试中因为模板展开等因素，出现了原本的输入处理需要添加大量接口识别、调用栈和使用点追溯相关的代码，且工程量极大。手写IR测试通过但端到端失败，根本原因是测试分层缺失——功能性测试验证了算法设计，但没有回归测试来固化现实世界的复杂性。这反映了我在测试程序经验上的不足，也导致开发信心受挫。

目前，我已经用这种我想克服的开发模式开发了规模可观的代码，且验证了一些模式，因此不可能放弃既有的仓库。

## 二、核心目标

我希望基于LLVM等成熟项目的Commit/PR规范来对我现有的AI辅助软件开发流程进行约束，将Agent的大Commit拆解，打磨每个小Commit的品质——分解功能实现和测试用例，让它符合独立验证等开发原则。最终目标是：**不是让AI写出更多代码，而是让自己通过AI的代码，持续增进对项目的理解和掌握。**

## 三、方法论基础：工业级项目的Commit/PR规范

基于LLVM、GCC、Linux Kernel、Triton、IREE等项目的Contributor Guideline，提炼出以下可落地的规范，完整原则见 `./COMMIT_PRINCIPLES.md`：

| 规范项 | 具体要求 | 来源项目 |
| :--- | :--- | :--- |
| **原子性（单一职责）** | 一个Commit只做一件事，禁止混入无关重构或代码清理 | 全部项目 |
| **规模上限** | 单次变更不超过200行。LLVM平均约220行，>500行即被视为"大型"提交 | LLVM |
| **可独立验证** | 每个Commit必须可独立编译和测试，确保库的每一次提交都处于可工作状态 | LLVM, GCC |
| **强制附带测试** | 所有代码变更必须同时提供对应的测试用例（正面、负面、边界场景） | LLVM, GCC, Linux |
| **提交信息规范** | 标题行≤75字符，以"组件:"开头；正文必须清晰回答"做了什么"、"为什么做" | Linux, GCC |


## 四、测试分层体系

完整原则见：`TESTING_PYRAMID.md`

| 测试类型 | 定位 | 存放位置 | 维护策略 |
| :--- | :--- | :--- | :--- |
| **功能性测试** | 开发者设计的"契约"，定义Pass应该做什么。IR简洁、教学式，是对功能行为的**定义** | `tests/features/` | 可随设计演变而修改，是"活的" |
| **回归测试** | 从真实SYCL/模板展开代码中用工具提取的IR片段，固化已修复的Bug。IR可能复杂怪异，是对现实复杂性的**固化** | `tests/regressions/` | 绝不随意修改，是"死的"但绝不能丢 |
| **端到端测试** | 完整的真实应用程序，编译并运行以检验正确性和性能 | 独立仓库或CI定时任务 | 周期性运行，作为最终安全网 |

**关键原则：功能性测试是活的规范，回归测试是死守的防线。两种测试必须共存且分类管理。**

## 五、双仓库架构

### 5.1 架构设计

```
AI快速交付库 (project-ai-lab)          精品手工库 (project)
─────────────────────────────          ───────────────────────
定位：试验场与素材库                    定位：知识库与艺术品
目标：快速看见端到端路径可行，增强信心    目标：凝聚个人理解和洞见，确保真正掌握代码
允许：大Commit、快速迭代、推倒重来       要求：小Commit（原则上≤200行）、规范测试、清晰提交信息
操作：AI自由生成                        操作：人工挑选移植，绝不直接运行无约束AI
```

### 5.2 代码流向

**单向流动**：从AI库"提取"和"移植"到精品库。移植的最小单元必须是"测试用例 + 核心逻辑"——先从AI功能块中提取回归测试（`.ll`文件等）提交到精品库，再根据测试手工重新实现或严格约束AI实现核心逻辑，确保每一行进入精品库的代码都被完全理解。

### 5.3 仓库关联

每当一个功能成功移植，在AI库的最后一次提交上打标签（如 `transplanted/feature-name-v1`），作为连接两个仓库的"航海日志"。

## 六、工具链设计

### 6.1 半自动IR缩减工具链

不手写复杂IR，不直接修改真实应用，而是通过工具流水线提取最小化测试用例：

| 工具 | 角色 | 使用阶段 |
| :--- | :--- | :--- |
| **llvm-extract** | 从真实程序的IR文件中提取目标函数，做第一步粗筛（可立即将数万行减少到数百行） | 第1步 |
| **bugpoint + llvm-reduce** | 自动缩减IR，锁定触发Bug的最小片段。`llvm-reduce`激进缩减后用`bugpoint`进一步优化 | 第2步 |
| **验证脚本 (test.sh)** | 用 `opt` + `FileCheck` 精确定义什么是"有趣的"变更，作为缩减过程的驱动标准 | 全程 |
| **手动迭代** | 在工具基础上做最终精简、重命名变量、添加注释，使测试用例可读 | 第3步 |

### 6.2 标准化提取流程

1. **精简模块**：使用 `bugpoint` 或 `llvm-extract` 将问题隔离到一个模块
2. **精简函数**：使用 `llvm-extract -func=your_func` 提取核心函数
3. **生成文本IR**：确保IR文件是文本格式（`.ll`），方便编辑和验证
4. **脚本驱动缩减**：编写验证脚本，以手动/半自动迭代方式删减无关代码，每轮删减后立即运行脚本验证
5. **微调与规范化**：重命名变量、删除无关元数据、添加注释

### 6.3 任务拆分与验证机制

**事前规划策略**：在Plan Mode中，不要求AI直接实现最终功能，而是先输出一个带验证检查点的实施计划。示例步骤：
- 步骤1：添加`.ll`测试用例，用FileCheck验证预期输出模式（可验证：`opt ... | FileCheck`通过）
- 步骤2：创建Pass骨架，注册到PassManager，确保能加载不崩溃（可验证：`opt ...`无错误退出）
- 步骤3：实现核心模式匹配，处理最常见的简单情况（可验证：步骤1的测试通过）
- 步骤4-N：逐步添加边界处理，每步新增对应测试

**事后拆分策略**：若已拿到大变更，用测试文件"切香肠"——先写出你认为是独立功能点的测试文件（它们当前必定失败），再从大代码块中逐步挑选出能让每个测试通过的最少代码，形成独立Commit。

## 七、文档策略

**核心原则：文档只记录"为什么"，不记录"怎么做"。** 后者由提交历史 + 测试用例承载——它们才是永不落灰的"活文档"。

| 文档类型 | 角色 | 维护成本 |
| :--- | :--- | :--- |
| **CLAUDE.md** | 项目宪法和工作流规范，固化从社区学到的Commit/PR/测试规范 | 低频更新 |
| **设计决策记录 (ADR)** | 记录为什么选择方案A而非方案B、权衡了什么。存入`docs/decisions/`，每次重大设计变更后由`/adr`命令生成。即使后来被推翻，也是宝贵的历史 | 只新增，不修改 |
| **测试目录说明** | 解释`tests/features/`和`tests/regressions/`的组织规则及运行方式 | 极简且高度稳定 |
| **README.md** | 唯一的入口指南：项目目标、如何构建、如何运行测试、`CLAUDE.md`在哪 | 低频更新 |
| **提交历史 + 测试用例** | 承载所有功能细节和演进历史，通过精心编写的小Commit和可执行测试构成 | 与代码同步 |
| **传统功能说明文档** | **建议放弃**。细节会被代码颠覆，用可执行的测试用例替代 | 零 |

## 八、人机分工原则

| 谁 | 职责 |
| :--- | :--- |
| **开发者（你）** | 定义"意义"：什么是一个逻辑单元、什么样的测试算通过、何时可以提交、何时应该转向。始终掌握对代码意义的最终解释权 |
| **AI** | 生成代码、草拟提交信息、分析代码结构、建议拆分方案、生成ADR和测试骨架、执行重复性检查和验证 |

**核心理念：不是"AI替你工作"，而是"AI为你工作"。拆分粒度取决于"意义"，而意义只有人才能定义。Agent可以帮你把1000行代码拆成10个100行的提交，但它无法知道哪一行代码是核心算法骨架，哪一行是性能微调——把知识从代码中蒸馏出来的，是人阅读代码时的思考。**

## 九、转向和三次原则

当为适配真实IR而不断打补丁、核心算法变得臃肿时，这是**转向信号**。遵循"三次原则"：

- **第一次实现**：验证思路可行（可能很粗糙）
- **第二次实现**：覆盖更多真实场景（开始积累技术债务）
- **第三次实现**：将前两次的经验整合进干净的设计

所有转向和重构的安全网是**测试套件**——无论内部如何重构，只要所有功能性测试和回归测试通过，就有坚实的信心。转向时保留所有测试用例，它们是新方案正确性的"验收标准"。将过去尝试、遇到的问题、最终决策的原因记录在ADR中，作为未来决策的参考。

## 十、预期收益

1. **开发信心**：AI库快速验证端到端可行性，不因工程细节而丧失对核心算法的信心
2. **深度掌握**：手工库的小Commit拆解过程本身就是最高效的"费曼学习法"实践——每一条精心编写的提交信息都是向未来自己解释"为什么这么做"的最佳文档
3. **可回溯性**：每个Commit附带测试且可独立验证，任意节点都可回退和审计
4. **论文复现能力**：这套方法论可直接应用于解剖5000~30000行的系统论文代码（如PPoPP、ASPLOS论文），将其重构为可理解的MVP和工程trick列表
5. **持续演进**：即使项目停滞一段时间，回归测试和ADR能快速恢复上下文，手动维护过程中的"留白"本身就是新idea的孵化器
6. **能力跃迁**：从"使用工具的人"进阶为"构建系统的人"，将AI的输出转化为自己的认知

## 十一、预期Artifact

我希望这是一套基于claude code的软件开发套件，可复用于任何仓库，辅助研究生/博士生的代码学习过程。

### 1、基础配置类

1. `CLAUDE.md`：固化Commit规范（≤200行、原子性）、测试分层策略、文档策略、人机分工原则，以及为了达到对应规范所需的工具

### 2、自动化防线类（无感的规则守卫）

1. **超大Commit拦截Hook**：配置 `PreToolUse` Hook，在 `git commit` 执行前检查暂存区变更行数，超过200行C/C++文件直接阻止，并给出拆分建议
2. **任务结束测试提醒Hook**：配置 `Stop` Hook，每次Agent任务结束后，主动提醒运行回归测试，避免“改了代码忘了跑测试”，或者能够启动subagent运行测试


### 3、可复用命令/技能类


1. **Commit历史分析与拆分Skill**：查询AI快速交付库或者想要学习的仓库的commit历史，分析代码结构，建议如何将大Commit拆分为可独立验证的小Commit
2. **仓库考古slash命令**：分析仓库代码结构，输出模块关系、入口函数、建议的Commit拆分演进路径
3. **测试生成Skill（含抽象降级警戒）**：按四象限（正面/负面/边界/正交）生成测试；当检测到手写IR过于“理想化”时，主动提醒是否存在模板展开、碎片化调用等真实IR才有的情况
4. **Plan前检查slash命令**：在把任务交给Agent之前，依据当前上下文检查任务粒度，如果过于宏大则提出分解建议并输出带检查点的分步计划。用于手工库的“警戒哨”
5. **贴身Mentor Skill** ：回答LLVM/MLIR API、CUDA/SYCL规范、设计模式、编译器开发最佳实践等方面的问题，减少查阅资料的时间。可配置为自动识别技术问题并介入
6. **学习笔记生成skill**：读取被学习的仓库（应当是）的完整commit log（或设置一个截断数量），生成包含核心原理、关键设计决策、工程trick及对应代码位置的学习文档
7. **微任务执行slash命令**：约束AI完成单一原子任务（≤150行），完成后列出变更但不自动提交。配合事后拆分策略使用
8. **ADR生成slash命令**：一键生成设计决策记录，存入 `docs/decisions/`。重大设计变更后使用，记录“为什么这么做”

### 4、辅助脚本模板类

1. **IR兴趣测试脚本生成工作流** 利用 `llvm-reduce`, `llvm-extract`, `bugpoint` 工具，结合`opt` + `FileCheck`制作兴趣测试，做到能够从真实IR中提取简化但能反映真实情况的回归测试用例。职能是缩减手工提取IR验证标准的时间
