# 预期Artifact分析：功能边界、形式建议与补充

对 `Motivation.md` 第十一部分的逐项分析。

---

## 前置说明：Claude Code 中的形式分类

在 Claude Code 中，"slash command"和"skill"在技术实现上是同一个机制（都是 `.claude/skills/` 目录下的 `SKILL.md` 文件，通过 `/name` 调用）。但**概念上可以区分**：

| 概念 | 特征 | 适用场景 |
|------|------|----------|
| **Skill** | 加载大量上下文知识，引导 AI 完成复杂的多步推理 | 需要专业领域知识的任务（如测试生成、API 问答） |
| **Slash Command** | 触发明确、边界清晰、一次性完成的操作 | 约束性操作（如 ADR 生成、微任务执行） |
| **Hook** | 事件驱动、自动触发、无需手动调用 | 拦截/提醒类（如超大 commit 拦截、测试提醒） |
| **CLAUDE.md 指令** | 始终加载在上下文中，定义行为约束 | 全局规范（commit 规范、测试策略） |
| **Shell 脚本** | 独立可执行文件，AI 可以调用但不依赖 AI | 确定性操作（如 IR 缩减流水线） |

### 选择原则

```
需要领域知识引导多步推理        → Skill
需要一键触发、边界明确           → Slash Command（轻量 Skill）
需要自动拦截/提醒                → Hook
需要始终生效的行为约束           → CLAUDE.md
需要确定性、可脱离 AI 独立运行   → Shell 脚本
```

## 逐项分析

### 类别1：基础配置类

---

#### 1. CLAUDE.md

**功能边界定义：**

- **职责（做什么）**：定义项目的"宪法"——行为规范、工作流约束、质量标准。它是所有 Agent 会话的默认上下文。
- **不做什么**：不执行操作、不替代 Skill 的领域知识、不记录具体的技术细节（技术细节交给 ADR 和测试用例）。
- **输入**：无（始终加载）
- **输出**：约束 AI 行为
- **作用范围**：当前项目所有会话

**内容建议（按优先级排列）：**

```
1. Commit 规范（≤200行、原子性、message 格式）
2. 测试分层策略（features/ vs regressions/ 的规则）
3. 人机分工原则（谁定义意义、谁生成代码）
4. 双仓库架构与代码流向
5. 文档策略（什么该写、什么不该写）
6. 可用工具/命令清单（指向其他 artifact 的索引）
7. 禁止事项（如：不允许在手工库中直接运行无约束 AI）
```

**形式：`CLAUDE.md`**（项目根目录，Claude Code 原生机制）

**与其他 artifact 的关系：** CLAUDE.md 是"索引"——它不重复 Skill 的详细指令，但告诉 AI 在什么情况下应该调用哪个 Skill。例如：

```markdown
## 可用命令

- `/micro` — 执行单一原子任务（≤150行），完成后不自动提交
- `/adr` — 生成设计决策记录
- `/review-before-commit` — 提交前的自检清单
- ...
```

### 类别2：自动化防线类（Hook）

---

#### 2. 超大Commit拦截Hook

**功能边界定义：**

- **职责**：在 `git commit` 执行前，检查暂存区中 C/C++/Python 源文件的净增行数。超过阈值（200行）时阻止提交，给出拆分建议。
- **不做什么**：不检查非代码文件（如 `.md`, `.txt`）、不阻止 revert 操作、不替代 Code Review。
- **触发条件**：`PreToolUse` hook，匹配工具 `Bash`，匹配命令 `git commit*`
- **输入**：`git diff --cached --stat` 的输出
- **输出**：允许提交 / 阻止 + 提示信息
- **作用范围**：仅手工库（精品库），AI 库不启用

**边界场景处理：**

```
允许放行的情况：
  - revert commit（git revert 产生的 commit）
  - 仅涉及 .md / .txt / 配置文件 的 commit
  - 删除代码为主的 commit（净增为负）

需要阻止但特殊处理：
  - 自动生成的文件（如 configure: Regenerate）→ 不计入行数
  - 测试数据文件（.ll 回归测试）→ 可配置更高阈值
```

**形式：`Hook`**（`settings.json` 中的 `PreToolUse` 配置）

**配置位置：** `.claude/settings.local.json`（项目级，不提交到仓库）

**伪代码逻辑：**

```bash
# 1. 检查是否是 git commit 命令
# 2. 运行 git diff --cached --numstat
# 3. 过滤非代码文件，计算净增行数
# 4. 如果 > 200，阻止并输出：
#    "暂存区包含 X 行变更（阈值 200）。建议：
#     - 用 /micro 命令拆分此提交
#     - 将测试文件分离为独立 commit
#     - 如果是自动生成文件，用 --skip-hook 跳过"
# 5. 否则放行
```

---

#### 3. 任务结束测试提醒Hook

**功能边界定义：**

- **职责**：每次 Agent 任务结束（或对话回合结束）后，检查是否有未提交的代码变更，如果有，提醒运行测试。
- **不做什么**：不自动运行测试（可能耗时很长），不替代 CI。它是"提醒"，不是"强制执行"。
- **触发条件**：`Stop` hook
- **输入**：`git status --short` 检查是否有未提交变更
- **输出**：有变更 → 提醒 + 询问是否启动 subagent 跑测试；无变更 → 静默

**形式：`Hook`**（`settings.json` 中的 `Stop` 配置）

**建议分为两档：**

```
轻量模式（默认）：
  - 仅提醒："你有未提交的变更，记得运行测试"
  - 输出可用测试命令

深度模式（可配置）：
  - 提醒 + 自动启动 subagent 运行 `make check` 或等效命令
  - 测试结果直接推送给用户
```

#### 4. Commit历史分析与拆分Skill

**功能边界定义：**

- **职责**：接收一个大 commit 的 hash（或 PR 编号），分析其 diff 内容，输出拆分方案——建议将这个大 commit 拆分为哪些小 commit，每个小 commit 包含哪些文件/函数变更，以及每个小 commit 的独立验证方式。
- **不做什么**：不自动执行拆分。拆分决策始终由人确认。
- **输入**：commit hash 或 PR URL
- **输出**：结构化的拆分方案（JSON 或 Markdown），包含：
  - 每个子 commit 的建议标题
  - 包含的文件列表
  - 独立验证方法
  - 子 commit 之间的依赖顺序
- **适用库**：AI 库（拆分后移植到手工库）

**形式：`Skill`**

原因：需要加载大量上下文（Commit/PR 规范、测试分层策略）并引导多步推理。

**使用示例：**

```
/split-commit abc123f
```

#### 5. 仓库考古slash命令

**功能边界定义：**

- **职责**：分析一个仓库（可以是本地目录或 GitHub URL）的代码结构，输出模块关系图、入口函数、关键数据结构，以及建议的阅读/学习路径（按 commit 演进顺序）。
- **不做什么**：不修改代码，不执行测试，不替代 `git log`。
- **输入**：仓库路径或 GitHub URL
- **输出**：
  1. 模块拓扑图（文字 ASCII 或 mermaid）
  2. 入口函数清单（main / Pass::runOnFunction 等）
  3. 建议的 commit 阅读顺序（"先读这个 commit 理解数据结构，再读那个 commit 理解算法..."）
- **适用场景**：接手新仓库、学习论文代码

**形式：`Skill`**（或轻量 Skill = Slash Command）

原因：需要探索仓库结构、分析 git 历史，属于一次性触发、边界清晰的操作。

**命令名建议：** `/archeology` 或 `/repo-map`

#### 6. 测试生成Skill（含抽象降级警戒）

**功能边界定义：**

- **职责**：给定一个 Pass 或功能描述，按四象限生成测试骨架（正面/负面/边界/正交）。**关键能力**：检测手写 IR 是否过于"理想化"（如缺少模板展开后的碎片化调用、无真实 metadata、无异常控制流），并在检测到时发出"抽象降级警报"。
- **不做什么**：不修改被测代码本身。生成的测试是骨架，需要人确认后精修。
- **输入**：Pass 源码 + 功能描述 + 可选的一段手写 IR
- **输出**：
  1. 四象限测试文件（`.ll` + FileCheck 指令）
  2. 如果有抽象降级风险：警报 + 建议（如"请运行 IR 缩减工具提取真实 IR"）
- **适用库**：手工库和 AI 库均可使用

**形式：`Skill`**

原因：测试生成是编译器领域专业任务，需要四象限方法论、FileCheck 语法知识、抽象降级检测逻辑。

**抽象降级检测信号：**

```
危险信号（触发警报）：
  □ IR中所有基本块都是单一前驱单一后继（过于规整）
  □ 无任何 alloca/load/store（栈操作缺失）
  □ 无任何 @llvm.* 内部调用（无内置函数调用）
  □ 函数调用全为直接调用，无间接调用
  □ 无任何 !dbg / !alias.scope 等 metadata
  □ 类型系统过于简单（无嵌套结构体、无指针的指针）
  □ 分支条件全为简单比较，无复杂条件表达式

安全信号（不触发）：
  ✓ 包含来自真实 SYCL/模板展开的 IR 片段
  ✓ 存在多层嵌套的指针/结构体类型
  ✓ 包含非平凡的 debug metadata
```

#### 7. Plan前检查slash命令

**功能边界定义：**

- **职责**：在进入 Plan Mode 之前，分析当前需求的粒度。如果需求过于宏大（预估变更 > 400 行或需要 ≥ 5 个独立步骤），阻止进入 Plan Mode 并给出分解建议。
- **不做什么**：不替代 Plan Mode 本身。它是在 Plan Mode 之前的"粒度审查"。
- **输入**：需求描述文本
- **输出**：
  - 粒度评估（合适 / 过大）
  - 如果过大：分解为 2-4 个子需求的建议，每个带预估行数
  - 如果合适：放行，进入 Plan Mode
- **适用库**：仅手工库

**形式：`Slash Command`**（轻量 Skill）

原因：逻辑简单——评估粒度 → 通过或分解。不需要深度推理。边界清晰，一次性操作。

**命令名建议：** `/pre-plan` 或 `/check-scope`

#### 8. 贴身Mentor Skill

**功能边界定义：**

- **职责**：回答 LLVM/MLIR/CUDA/SYCL/编译器设计相关的技术问题。关键是限定领域——它不是一个"什么都懂"的通用问答，而是编译器开发领域的专业 mentor。
- **不做什么**：不写代码、不修改文件、不替代 Google/官方文档（而是帮助理解和定位）。
- **输入**：技术问题（自然语言或代码片段）
- **输出**：解释、API 示例、文档链接、设计建议
- **作用范围**：当检测到技术问题时自动介入 / 或通过 `/mentor` 手动调用

**形式：`Skill`**

原因：需要加载领域知识（LLVM Programmer's Manual、MLIR Dialect 文档、设计模式等）。

**重要边界约束：**

```
该 Skill 必须有明确的知识范围声明：

涵盖：
  ✓ LLVM IR/MLIR 的 API 与 Pass 开发
  ✓ CUDA/SYCL 编程模型与优化
  ✓ 编译器设计模式（Visitor, Listener, Pass Pipeline）
  ✓ 调试技巧（Pass crash 分析、opt 工具链）

不涵盖：
  ✗ 通用编程问题（Python/Rust/Go 语法）
  ✗ 非编译器领域的算法设计
  ✗ Claude Code 使用问题（此由官方文档覆盖）

不确定时：诚实说"我不确定"，而非猜测
```

**自动介入条件：**

```
检测用户消息中包含以下模式时主动介入：
  - "为什么这段 IR ..."
  - "这个 Pass 应该..."
  - "opt 报错 ..."
  - "@llvm.* 怎么用"
  - "llvm::" / "mlir::" API 调用
```

#### 9. 学习笔记生成skill

**功能边界定义：**

- **职责**：读取目标仓库的 commit log（可配置截断数量，默认最近 200 条），从中提取核心原理、关键设计决策、工程 trick 及对应代码位置，生成一份结构化的学习文档。
- **不做什么**：不修改目标仓库。生成的笔记是"学习辅助"，不是"自动生成的文档替代品"。不替代人阅读代码的过程。
- **输入**：目标仓库路径 + commit 范围（默认 HEAD~200..HEAD）
- **输出**：Markdown 学习笔记，包含：
  1. 项目架构概览（自动提取模块关系）
  2. 关键 commit 列表（按主题分组，不是按时间）
  3. 核心算法/数据结构的演进轨迹
  4. 工程 trick 摘录（带代码引用）
  5. 推荐阅读顺序

**形式：`Skill`**

原因：需要分析 git 历史、理解代码变更语义、生成结构化笔记，属于复杂的多步推理。

**使用示例：**

```
/learn-from ~/codebase/llvm-project  # 分析 LLVM 仓库
/learn-from ~/codebase/triton --commits 100 --focus "backend"
```

#### 10. 微任务执行slash命令

**功能边界定义：**

- **职责**：约束 AI 完成单一原子任务（≤150行变更），完成后列出变更摘要但不自动提交。这本质上是一个"行为约束器"——它覆盖了 CLAUDE.md 中的默认行为，强制执行最严格的原子性。
- **不做什么**：不拆分任务（拆分由 `/split-commit` 或 `/pre-plan` 完成），不自动提交。
- **输入**：一个明确的、单一的任务描述
- **输出**：
  1. 代码变更（≤150行）
  2. 变更摘要（文件列表 + 行数 + 简要说明）
  3. 建议的 commit message
  4. 提醒："变更已完成，请审查后手动提交"
- **适用库**：仅手工库

**形式：`Slash Command`**（轻量 Skill）

**命令名建议：** `/micro`

**与 CLAUDE.md 的关系：**

```
CLAUDE.md 中声明全局规范："每个 commit ≤ 200 行，每次任务聚焦单一目标"
/micro 命令 = 在 CLAUDE.md 约束基础上的"最高警戒模式"
             → 行数上限降到 150
             → 禁止跨文件重构
             → 完成后强制不提交
```

#### 11. ADR生成slash命令

**功能边界定义：**

- **职责**：读取当前分支的变更内容，生成一份设计决策记录（ADR），存入 `docs/decisions/`。记录内容包括：标题、状态、背景、决策、备选方案、后果。
- **不做什么**：不做决策——只记录已完成的设计变更。不修改代码。
- **输入**：当前分支的 diff（或最近 N 个 commit）
- **输出**：`docs/decisions/NNNN-title.md`，遵循标准 ADR 格式
- **触发时机**：重大设计变更完成后，手动调用

**形式：`Slash Command`**（轻量 Skill）

**命令名建议：** `/adr`

**ADR 模板：**

```markdown
# ADR-NNNN: Title

## Status
[Proposed / Accepted / Deprecated / Superseded by ADR-XXXX]

## Context
为什么需要做这个决策？技术背景是什么？

## Decision
做了什么决策？具体方案是什么？

## Alternatives Considered
考虑过哪些替代方案？为什么没选？

## Consequences
这个决策带来了什么后果（正面和负面）？需要跟进什么？
```

### 类别4：辅助脚本模板类

---

#### 12. IR兴趣测试脚本生成工作流

**功能边界定义：**

- **职责**：提供一个完整的、可复用的工作流（Skill + 脚本模板），从真实 IR 中提取简化但反映真实情况的回归测试用例。工具链为 `llvm-extract → llvm-reduce/bugpoint → opt + FileCheck 验证脚本`。
- **不做什么**：不是一键全自动——缩减过程需要人的判断（什么算"有趣的"行为）。工具链辅助人做裁剪，但最终精简和命名由人完成。
- **输入**：
  - 完整程序的 IR 文件（`.ll`）
  - 目标函数名
  - 兴趣测试脚本（定义什么是"有趣的"行为）
- **输出**：
  - 最小化的 `.ll` 测试文件
  - 对应的 FileCheck 断言
  - 缩减过程的日志
- **适用库**：AI 库提取 → 手工库沉淀

**形式：`Skill` + `Shell 脚本模板`**

原因：
- Skill 部分：引导 AI 使用工具链，解释每一步的作用
- 脚本模板部分：`test.sh` 模板（`opt + FileCheck`），可脱离 AI 独立运行

**工作流步骤（Skill 引导）：**

```
Step 1: llvm-extract --func=target_func input.ll -o extracted.ll
        （从数万行中粗筛到数百行）

Step 2: 编写兴趣测试脚本 test.sh
        #!/bin/bash
        opt -passes=your-pass extracted.ll -S | FileCheck extracted.ll
        （定义什么是"有趣的"行为）

Step 3: llvm-reduce extracted.ll --test=test.sh -o reduced.ll
        （自动缩减到最小复现）

Step 4: 手工精修
        （重命名变量、添加注释、确认断言准确）

Step 5: 将最终测试放入 tests/regressions/
```

## 补充建议：缺失的 Artifact

以下是我认为当前清单中缺失的 artifact，按优先级排列。

### 补充1：移植助手 Skill（高优先级）

**对应 Motivation.md 第五章"双仓库架构"**

当前清单中，AI库→手工库的移植流程没有任何工具支持，完全靠人。建议：

**功能边界：**
- 输入：AI 库中的一个 commit/PR
- 自动提取其中的测试用例（`.ll` 文件等）
- 分析哪些代码变更被测试覆盖、哪些没有
- 输出：移植清单（测试 → 核心逻辑 → 边界处理，按依赖顺序排列）
- 移植完成后，在 AI 库中打标签 `transplanted/feature-name-v1`

**形式：`Skill`**，命令名建议 `/transplant`

### 补充2：Commit Message 格式校验 Hook（高优先级）

**对应 COMMIT_PRINCIPLES.md 规则 4**

当前清单有"超大 Commit 拦截"但缺少"Commit Message 格式检查"。

**功能边界：**
- 在 `git commit` 前检查 message 格式
- 标题 ≤ 75 字符
- 包含组件标签（如 `[PassName]`）
- 标题与正文间有空行
- 正文解释了 WHY
- 没有 `@` 提及

**形式：`Hook`**（`PreToolUse`，匹配 `git commit`）

### 补充3：测试覆盖缺口分析 Skill（中优先级）

**对应 TESTING_PYRAMID.md 原则 2**

**功能边界：**
- 输入：一个 commit 或 PR
- 检查变更的代码文件是否都有对应的测试文件变更
- 检查新增函数是否都有测试覆盖
- 输出：覆盖缺口报告 + 建议补充的测试类型

**形式：`Skill`**，命令名建议 `/coverage-gap`

---

### 补充4：转向检测与提醒 Skill（中优先级）

**对应 Motivation.md 第九章"转向和三次原则"**

**功能边界：**
- 监测当前的开发模式是否出现"转向信号"：
  - 同一 Pass 中补丁式修改超过 N 次
  - 核心算法文件的行数持续膨胀
  - `.ll` 测试文件中为适配真实 IR 不断增加分支条件
- 检测到信号时主动提醒：建议评估是否需要转向（第三次实现）
- 输出：转向评估报告（当前状态 vs 第一次/第二次实现的差异分析）

**形式：`Skill`**（可设为定期自动触发或手动调用 `/check-turn-signal`）

---

### 补充5：Git Bisect 辅助 Skill（低优先级）

**对应测试分层策略中的回归测试定位**

**功能边界：**
- 输入：一个已知通过和一个已知失败的 commit hash
- 自动执行 `git bisect`，在每一步运行指定的测试命令
- 定位引入失败的精确 commit
- 输出：commit hash + diff + 建议的修复方向

**形式：`Skill`**，命令名建议 `/bisect`

---

### 补充6：pre-commit 核查表 Skill（低优先级）

**对应 COMMIT_PRINCIPLES.md 第四章核查表**

**功能边界：**
- 在准备提交时，逐项检查核查表的 10 个条目
- 每一项给出通过/未通过 + 具体建议
- 全部通过后，生成建议的 commit message

**形式：`Slash Command`**，命令名建议 `/pre-commit`
