### 📜 文档的核心角色：只记录“为什么”，不记录“怎么做”

对于你这种快速迭代的研究型项目，文档应该是一个**决策日志**和**意图档案**，而不是一份**用户手册**。

| 文档类型 | 角色定位 | 是否需要维护 | 应对过时的策略 |
| :--- | :--- | :--- | :--- |
| **设计决策记录 (ADR)** | 记录**为什么**选择方案A而不是方案B，权衡了什么。 | 是，这是核心资产 | 永不更新，只新增。每个决策是一个文件，即使后来被推翻，也是宝贵的历史。 |
| **测试目录说明** | 解释 `tests/features/` 和 `tests/regressions/` 的组织规则，以及如何运行。 | 是，但极简 | 内容高度稳定，很少变动。 |
| **贡献/工作流指南** | 固化你从社区学到的 Commit/PR 规范，供 AI 和未来的你遵循。 | 是，但演变慢 | 它就是你的“项目宪法”，建议放入 `CLAUDE.md`。 |
| **架构概览** | 一张“地铁图”，标明主要模块和它们之间的数据流。 | 低频更新 | 只在模块拆分/合并时更新，用 Mermaid 等图表，AI 可轻松生成。 |
| **代码注释/提交信息** | 解释**怎么做**和**做了什么**。 | 与代码同步 | 这是你唯一的“实时文档”。通过良好的提交历史（小Commit + 规范信息）构成。 |
| **传统功能说明文档** | ❌ **建议放弃**。它的细节100%会被代码颠覆。 | 不维护 | 用可执行的测试用例（`.ll` + `FileCheck`）替代。测试就是你的功能规格说明书。 |

> **核心原则**：把“文档”从过去时的“总结”变成现在时的“意图记录”。**提交历史 + 测试用例**就是你永不落灰的活文档。

### 🧰 如何与你的工作流结合，让文档不费力

你已经有了一个强大的 AI 协作工作流，可以直接把文档更新嵌入进去。

#### 1. 让 `CLAUDE.md` 成为你的“项目宪法”
我们已经为它制定了 Commit 规范、测试法则。你还可以加入一条：
```markdown
## 文档规范
- 设计决策记录在 `docs/decisions/` 目录下，命名格式为 `YYYYMMDD-简短标题.md`。
- 每次重大设计变更后，AI 必须草拟一份决策记录，供我审核。
- 不对函数、类编写孤立的功能说明文档，所有功能细节应由测试用例和提交信息承载。
```

#### 2. 用自定义命令 `/adr` 零成本生成决策记录
在 `.claude/commands/adr.md` 中创建：
```markdown
请根据我们当前的会话上下文，为我草拟一份设计决策记录（ADR），存入 `docs/decisions/` 目录。
内容包括：
1. **背景**：我们遇到了什么问题？
2. **决策**：选择了什么方案？
3. **后果**：带来了哪些便利，又留下了哪些待解决的问题？
```
每当你和 AI 共同完成一个非平凡的模块重构或方案选择，只需输入 `/adr`，AI 就会帮你生成一份结构化文档。你审核后提交即可。这份记录的价值在未来会越来越大。

#### 3. 利用 Git 历史作为“活文档”
你的小Commit规范会自然地产生一条信息密度极高的提交日志。未来你想知道某个功能为什么变成现在这样，不需要去翻文档，直接：
```bash
git log --grep="<模块名>"
```
就能看到一系列精心编写的小Commit，每个都解释了“做了什么”和“为什么”。这是对抗文档过时的终极武器。
