# ADR-0001: 拆分-分解-微执行 三层架构

## Status

Accepted (2026-05-24)

## Context

v0 版本的 `/split-commit` 承担了过多职责：从 commit 全景分析一路做到 ≤200 行子 commit 的精细拆分，同时还要加载 COMMIT_PRINCIPLES.md 约束。实测中输出质量不稳定——模块依赖分析、原则校验、测试匹配、粒度对齐全压在一个 skill 中，每一步的决策质量被稀释。

此外，用户反馈了三个具体问题：
1. 拆分后的子 commit 可能将"实现"和"测试"分为两个独立提交，违反独立验证原则
2. 大型函数（>200 行）在多情况分支下的分解缺乏工具支持
3. 原则加载时机过早（在模块结构厘清之前），干扰结构分析

## Decision

将原 `/split-commit` 拆分为三个职责清晰的 Skill：

```
/split-commit      模块级骨架分析
      ↓
/decompose         文件组内细粒度分解（新增）
      ↓
/micro             原子执行 + 测试匹配 + 独立验证
```

### 职责边界

| Skill | 输入 | 输出 | 加载原则？ |
|-------|------|------|-----------|
| `/split-commit` | commit hash | 模块骨架 + include 耦合聚类 + 建议 `/decompose` 调用列表 | **否** |
| `/decompose` | 文件组 (.h+.cpp+CMake) | 子 commit 序列 (接口→算法→边界)，含超级函数分支分类 | **否** |
| `/micro` | 单一原子任务描述 | 可独立验证的 commit (实现+测试) | **是** (COMMIT_PRINCIPLES.md) |

### `/split-commit` 变更

- **移除**：原则加载、细粒度拆分逻辑、测试匹配
- **新增**：`#include` 依赖分析（`git show <hash> -p | grep "^+#include"`）用于文件聚类
- **新增**：耦合度判断——同一模块内通过 include 关联的文件归为一个文件组，输出时标注每组建议调用一次 `/decompose`
- **保留**：`--name-only` 路径扫描、模块检测、多模块依赖询问

### `/decompose` 新增

- **输入**：一个文件组（如 `MyPass.h` + `MyPass.cpp` + 对应 `CMakeLists.txt`）
- **输出**：分阶段实现计划（接口定义→核心算法→正常路径→边缘路径→错误处理），每阶段带行数估算和测试入口建议
- **超级函数处理**：基于控制流节点（条件判断、switch、return）进行分支分类；可分的标注置信度，纠缠的标注"不确定"并建议人工截断
- **粒度下限**：若文件组 ≤40 行且结构简单，直接报告"无需分解，可直接交 `/micro`"

### `/micro` 变更

- **新增**：测试匹配职责——根据约定（`compiler/lib/Foo/` ↔ `tests/foo/`）自动定位测试目录
- **新增**：测试修剪/补充——执行前先评估现有测试质量，标记缺口，必要时补充正/负/边界测试用例
- **保留**：≤150 行约束、不自动提交、变更摘要输出

### 原则加载策略

COMMIT_PRINCIPLES.md 仅由 `/micro` 加载——在模块结构已厘清、文件组已分解、原子任务已明确的时点。此时原则真正可以落地执行，而非提前约束干扰上游分析。

## Alternatives Considered

| 方案 | 优点 | 缺点 | 为何未选 |
|------|------|------|----------|
| 保持 `/split-commit` 单体 | 单一入口，用户无需学多个命令 | 输出质量随 commit 复杂度急剧下降 | 用户实测已确认 |
| `/split-commit` + `/micro` 两个（不引入 `/decompose`） | 少一个命令 | 文件组内细分的职责无处安放，`/micro` 负担过重 | 职责模糊 |
| `/decompose` 合并进 `/split-commit` | 减少调用链长度 | 回到职责过载问题 | 反模式 |

## Consequences

- **正面**：三个 Skill 的复杂度各自可控，输出质量预期提升；`/split-commit` 不再被原则约束，结构分析更专注
- **负面**：命令链从 2 个增加到 3 个（`split-commit → decompose → micro`），新增一条调用依赖；需要用户学习 `/decompose` 的用法
- **需要跟进**：`/decompose` 的超级函数分支分类算法在实际复现度项目上的准确率验证；AST Matcher 精确模式后续封装为 `scripts/` 下的工具
