# 微任务执行

## 职责

以最高行为约束等级执行单一原子任务，交付可独立验证的 commit。核心职责包括：

1. **加载 COMMIT_PRINCIPLES.md 作为硬约束**——确保每次输出符合测试完备性、独立验证、单一关注点
2. **自动测试匹配**——根据约定映射定位测试目录，评估现有测试质量，必要时补充正/负/边界测试用例
3. **原子实现**——净增 ≤150 行，不自动提交，不扩大范围

## 不做什么

- 不自动提交（`git commit` 始终由开发者手动执行）
- 不拆分任务（拆分交给 `/split-commit` 和 `/decompose`）
- 不扩大任务范围——发现"顺手可以改 X"时暂停询问
- 不跨文件重构（除非任务明确要求且总行数仍在 ≤150 行内）
- 不修改已有测试中的通过用例——仅补充缺失的测试，不动已有断言的正确逻辑

## 输入

- **必需**: 一个明确的任务描述（自然语言）
- **典型来源**: `/decompose` 输出的 Phase 计划中的一条，或 `/split-commit` 输出的文件组中标注"可直接交 /micro"的文件组

## 行为约束

```
CLAUDE.md 全局约束:
  - 每次任务聚焦单一目标
  - 每个 commit ≤ 200 行
  - 不自动提交

/micro 在此之上的强化:
  - 净增行数上限: ≤ 150 行（测试文件不计入）
  - 加载 COMMIT_PRINCIPLES.md 规则 2/3/4 作为硬约束
  - 自动匹配测试：定位 → 评估 → 标注缺口 → 补充
  - 禁止顺手重构无关代码
  - 完成后必须输出: 变更摘要 + 建议 commit message + 审查提醒
```

## 行为准则

以下四条规范约束 AI 在 `/micro` 中的编码行为。与"行为约束"互补——约束是量化上限，准则是决策原则。

### 1. 先想后写

- 有歧义时不猜测——列出可能的解读，让用户选
- 有更简单的方案时说出来，不沉默执行复杂方案
- 不清楚时停下来，指明困惑点，提问

### 2. 最简实现

- 只写解决问题所需的最小代码。不添加未被要求的功能
- 不为单次使用的代码创建抽象
- 不处理不可能发生的错误场景
- 如果 200 行能写成 50 行，重写

### 3. 精准改动

- 不"顺手改进"无关代码、注释、格式
- 不重构没坏的东西。匹配现有风格
- 发现无关的死代码时，告知用户但不自行删除
- 只清理自己的变更造成的 orphan（未使用的 import/变量/函数）

**测试豁免**：上述限制不适用于测试文件。为本任务补充缺失的测试用例（正/负/边界）是独立验证的硬性要求，不属于"顺手改进"。对已有测试的正确逻辑不做修改。

### 4. 目标驱动

- 不执行"让它能跑"这种弱目标。将每个任务转化为可验证的成功标准
- 多步骤任务先列出简要计划，每步带验证检查点

## 工作流

### Step 0: 加载项目规范

在编写任何代码之前，读取 `docs/Principle/COMMIT_PRINCIPLES.md` 中的以下核心规则。**违反任何一条的子 commit 方案都必须被拒绝**：

- **规则 2（测试完备性）**：所有代码变更必须附带测试。新功能没有测试 = 不合规。
- **规则 3（独立验证）**：每个 commit 必须可独立编译和测试。
- **规则 4（单一关注点）**：一个 commit 只做一件事。

### Step 1: 理解任务

首先将任务转化为可验证目标：用户说"添加验证"→ 翻译为"为无效输入编写测试，然后让测试通过"；说"修复 bug"→ 翻译为"编写复现测试，然后修复"。弱目标（"让它能跑"）是不可接受的——必须在开始前澄清。

明确以下要素（如果任务描述缺了某个，主动问）：

- 要修改/新增哪些文件？
- 要实现什么功能或修复什么问题？
- 任务来源中包含的测试入口建议（如 `/decompose` 的 Phase 输出）

### Step 2: 测试匹配

根据实现文件的路径，按约定映射定位测试目录：

```
约定映射规则（基于编译器项目的常规布局）:

  lib/Transforms/Scalar/        → test/Transforms/Scalar/
  lib/Analysis/                 → test/Analysis/
  compiler/lib/<Module>/        → tests/<module>/
  python/triton/<Module>/       → python/test/unit/<Module>/

未匹配到时:
  1. 询问用户测试目录位置
  2. 若用户也不知道：在 tests/ 下创建对应目录，并标注"测试目录未预先存在"
```

定位到测试目录后，评估现有测试质量：

```
评估维度:

  □ 正常路径覆盖: 是否测试了最常见的使用场景？
  □ 负面测试: 是否测试了不应该触发的情况？
  □ 边界条件: 是否测试了空输入、单元素、极值？
  □ 输出验证: 是否用 FileCheck 精确验证了输出格式？

评估结果:
  - 全部覆盖 → 使用现有测试，不做修改
  - 有缺口 → 标注缺口类型，在 Step 4 中补充测试
```

### Step 3: 预估规模

在编写代码前做粗略估算：

```
预估: 新增 ~XX 行 (实现), ~YY 行 (测试), 净增 ~ZZ 行 (不含测试)
```

- 若净增 > 150 行 → 拒绝，建议缩小范围
- 若净增 ≤ 150 行 → 继续

### Step 4: 执行

编写代码，遵循：

- 先补充测试（如果 Step 2 发现缺口），再写实现
- 测试覆盖正常路径 + 负面场景 + 边界条件
- 每次编辑聚焦单一功能点
- 不顺手格式化、重命名无关变量

### Step 5: 自检

代码完成后，输出前自行检查：

```
□ 净增行数是否 ≤ 150？
□ 每个新增函数都有对应的测试？
□ 测试覆盖了正常路径 + 至少一个边界或负面场景？
□ 无调试代码 (errs(), printf, FIXME)？
□ 无意外引入的空白变化？
□ 独立验证：能否用一条命令验证此 commit 的正确性？
```

若任一项未通过，修正后再进入 Step 6。

### Step 6: 输出变更摘要

格式固定如下：

```
变更摘要:

  文件:
    src/MyPass.cpp        +85 -12  (核心逻辑)
    tests/features/mypass/basic.ll  +45 (功能测试)

  净增: 73 行 (不含测试), 测试: 45 行

  测试覆盖:
    正常路径: ✓ basic.ll — 双层嵌套循环的规约匹配
    负面场景: ✓ basic.ll — 非归约循环的透传
    边界条件: ✓ empty.ll — 空输入不崩溃

  独立验证命令:
    opt -passes=my-pass tests/features/mypass/basic.ll -S | FileCheck tests/features/mypass/basic.ll

建议 commit message:
  ┌──────────────────────────────────────────────────────┐
  │ [MyPass] Add pattern matching for nested loops       │
  │                                                      │
  │ 识别嵌套循环中的归约模式，调用 ReductionPattern      │
  │ 进行替换。目前仅处理双层嵌套，三层以上标记 TODO      │
  │ 并原样返回。                                         │
  └──────────────────────────────────────────────────────┘

⚠ 变更已完成，请审查 diff 后手动提交。
   提交前建议运行: /commit-review
   提交后建议运行: /verify
```

## 边界情况

| 场景 | 行为 |
|------|------|
| 任务描述模糊 | 提问澄清。不猜测。 |
| 预估净增 > 150 行 | 拒绝。建议缩小范围或先 `/decompose`。 |
| 执行中发现需额外改动 | 暂停，给用户 A/B 选项。 |
| 约定映射找不到测试目录 | 询问用户。若也不确定，自动创建并标注。 |
| 现有测试已覆盖全部场景 | 直接复用，不修改已有测试。 |
| 现有测试不够好（缺边界或负面场景） | 补充新测试，不动已有测试的正确逻辑。 |
| 任务来自 /decompose 的纯数据结构 Phase | 跳过测试匹配，仅编译通过即可。 |
| 在 AI 库中被调用 | 提醒用户 `/micro` 适用于手工库。 |

## 与 CLAUDE.md 的关系

`/micro` 不替代 CLAUDE.md——它在 CLAUDE.md 全局约束之上叠加临时强化约束 + 原则加载 + 测试匹配。任务结束后回到默认行为级别。

## 与上游 Skill 的协作

```
/split-commit → 模块骨架 + 文件组清单
/decompose    → 文件组内细粒度分解 (Phase 计划 + 测试入口建议)
/micro        → 原子执行 + 加载原则 + 测试匹配 + 独立验证 commit
```

`/micro` 是三层漏斗的出口——它确保最终产出的每个 commit 可以独立站住。
